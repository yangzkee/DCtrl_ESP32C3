ESP32 循线小车固件说明

当前工程面向 ESP32-S3，控制一个通过串口接收运动指令的小车底盘，并读取一个通过 VGTI/UART 通信的八路红外循线模块。工程结构已经按 ESP32-C3 迁移做了分层，后续更换芯片时优先调整板级配置，不改控制逻辑。

所有已经确认的运动、通信、安全、调参和小程序策略统一登记在 DCar-Liner 策略登记表。车端状态机与运动分层图见 DCar-Liner 状态机说明。新增或修改策略时，优先更新这些网页。

需要连接的外设

  • USB-C 串口：用于首次烧录、日志和恢复。
  • 底盘串口：ESP32 向底盘发送速度和转向指令。
  • 八路红外循线模块串口：ESP32 读取循线状态。
  • 共地连接：ESP32、底盘控制器、循线模块必须共地。
  • 电平保护：若外设是 5V UART，需要加电平转换，ESP32 侧保持 3.3V 逻辑。
  • 独立稳定供电：电机供电和 ESP32 供电要避免互相干扰或反灌。
  • 无线调试通道：BLE 默认开启，用于 DHelper 小程序收发 PID 和速度档位；Wi-Fi 热点只作为按需开启的后备诊断和无线烧录入口。

当前默认接线

ESP32-S3 默认使用两个独立 UART：底盘为 UART1 GPIO17/GPIO18，循线模块为 UART2 GPIO15/GPIO16。这些是工程默认值，最终应按实际开发板原理图确认后固化。

ESP32-C3 当前 profile 为：底盘 UART1 TX GPIO4 / RX GPIO5，循线模块 UART0 TX GPIO2 / RX GPIO3。接线时交叉连接：ESP 的 TX 接外设 RX，ESP 的 RX 接外设 TX，并且必须共地。也就是底盘接 GPIO4 -> 底盘RX、GPIO5 <- 底盘TX；循线模块接 GPIO2 -> 模块RX、GPIO3 <- 模块TX。C3 控制台走 USB Serial/JTAG，UART0 保留给循线模块；如后续仍需换脚，只改 C3 板级 profile，不改业务模块。

当前巡线目标

本阶段地图只是一圈贴在地上的黑胶带。不规则来自手工贴胶带的自然误差，不是迷宫，也不是分岔路线。固件当前只做圆环巡线：等待启动、跟住线、丢线后线速度归零并用扩幅摆扫找线。迷宫、岔路识别、路线记忆和选路暂不进入主运动策略。

运动代码已经把硬件接线和底盘运动控制分开：line_trace_controller 负责读参数、读传感器、执行控制计划并发底盘指令；components/control 内部按 motion_contracts、motion_inputs、line_interpreter、motion_policy、motion_executor 分层。Layer 0 的 motion_contracts 是唯一真相层，集中保存黑白线语义、速度档位、偏移曲线、找线时间分段和限幅规则；control_plan 是策略层与执行层共享的控制指令合同，统一表达底盘动作和整车状态动作；输入层只整理和归一化数据；解析层估测小车与线路的几何关系；策略层生成运动意图；执行层转换成底盘命令输出合同。当前内部巡线相位会出现在遥测 controller.line_phase 中，例如 STOPPED、ACQUIRE_LINE、TRACK_LINE、LINE_LOST 和 SENSOR_FAULT。

当前协议选择

底盘串口采用 DFLink Motion_Velocity 指令，固件按固定循环频率持续发送 Vx 和 Vz。当前底盘语义中，Vx 是协议线速度值，Vz 是每条速度指令的角度增量，不是持续保持的角速度；正常循线和丢线找回都走这条速度指令路径。八路循线模块优先采用 USART 数字模式，固件发送 $0,0,1# 请求数字数据，并解析 $D,x1:0,...,x8:0# 这类返回帧。当前实测模块语义为黑线输出 0、白底输出 1，所以固件默认把 0 作为有效线。

八路模块虽然也支持 IO 和 I2C，但 IO 会占用 8 个 GPIO，不利于迁移到 ESP32-C3；I2C 需要进一步确认地址和寄存器协议。USART 数字模式当前协议最明确、占用引脚少、迁移风险低。

迁移到 ESP32-C3

ESP32-C3 的串口和引脚资源比 ESP32-S3 更紧张。工程已经把目标芯片差异集中在板级配置文件中；如果实际 C3 板卡无法同时提供两路独立 UART，只需要改板级 profile 或增加外部串口扩展，不需要重写底盘、循线和控制模块。

无线调试

固件启动后默认开启 BLE Low Energy 调试入口和 DCtrl 遥控串口入口，统一广播名为 DCtrl，不再通过小程序改名。调参 BLE 服务 UUID 为 7b3a0001-8d4d-4b9a-b5c7-0f7c4c415243，向 RX 特征 7b3a0002-8d4d-4b9a-b5c7-0f7c4c415243 写入命令，从 TX 特征 7b3a0003-8d4d-4b9a-b5c7-0f7c4c415243 读取响应。DCtrl 遥控串口服务 UUID 为 6d1f0001-5b9b-4f4e-9c67-7b1f4d430001，写入特征 6d1f0002-5b9b-4f4e-9c67-7b1f4d430001，通知特征 6d1f0003-5b9b-4f4e-9c67-7b1f4d430001。

DCtrl 遥控入口按旧蓝牙串口体验工作：小程序生成完整 DFLink 底盘速度帧，并在手机侧把业务层 Vz deg/cmd 转换为 DFLink 线上 rad/cmd；ESP32-C3 只把 BLE 写入字节原样转发到底盘 UART，不再解释速度、缓存目标、换算角度或用本地循环喂车。遥控成功路径保持静默，不再每个 BLE 写入回 OK，只有 E:* 错误会通知小程序。遥控坐标真相固定为 X > 0 前进、Y > 0 左平移、Z/Yaw > 0 左旋/逆时针，小程序负责把摇杆、按钮和体感输入映射进这个坐标系，固件透明桥不得再叠加符号修正。遥控占用 REMOTE_BRIDGE 模式时，巡线控制器不会启动、不会读传感器、不会发送后台停车帧；BLE 断开时固件只发送一次零速保护并回到 SAFE_IDLE。巡线控制器只在手动测试或自动巡线入口启动，并用 STARTING 状态阻止并发请求创建双任务。诊断 telemetry 中的 remote_bridge 字段只记录写入次数、字节数、当前遥控会话内最大写入间隔和错误计数，用于定位顿挫来源，不参与遥控运动策略。

DHelper 小程序的 line-tuning 功能主要使用短 BLE 帧调参：发送 G 读取参数，返回 P<kp>,<ki>,<kd>,<gear>；发送 S<kp>,<ki>,<kd>,<gear> 写入并保存参数；发送 N 读取固定蓝牙名 DCtrl；发送 N=* 清理旧自定义名称并恢复/确认 DCtrl；N=<name> 已禁用并返回 E:FIXED。当前可调范围固定为 Kp 7000-11000、Ki 0-300、Kd 0-3000、速度档位 1-3。速度档位会同时映射最大协议线速度和单次 Z 角度增量：一档最大协议线速度 250 mm/s（Vx=0.25）与 8000 mdeg/cmd，二档 600 mm/s（Vx=0.6）与 10000 mdeg/cmd，三档 1000 mm/s（Vx=1.0）与 10000 mdeg/cmd。自动循线时，检测到有效线逐渐偏离中间的 x4/x5 红外管，固件会用先慢后快的二次曲线降低线速度，偏到最外侧时大约保留当前档位 45% 的线速度；转向输入也使用归一化偏移的二次曲线，小偏离轻微修正，大偏离快速增大 Z 角度增量。如果 telemetry 中 motion.angular_mdeg_s 长时间贴近这些上限，说明当前主要受转向限幅影响，而不是单纯 PID 增益不足。

DHelper 小程序的循线控制面板使用 BLE JSON 命令：先点 一键自检，小程序依次读取 telemetry、必要时清除故障、执行 arm_auto，三步全部通过并进入 AUTO_ARMED 后才会点亮 开始循线。点击 开始循线 后发送 start_auto，车端进入 AUTO_RUNNING；面板中的 停止 会发送 stop。小程序源码已迁至 DHelper 小程序仓库，本仓库只保留车端协议和固件实现。

正式固件已经合入底盘诊断能力，不再需要单独维护临时探针固件。C3 当前底盘口为 TX GPIO4 / RX GPIO5 / 115200，接线仍然是 ESP32 的 TX 接底盘 RX、ESP32 的 RX 接底盘 TX。电脑端 BLE 工具可以发送 C? 查看底盘口配置，发送 Crepeat 连续查询 DFLink ready/version/params 并检查校验和，发送 Cxyz 测试一帧原始 Vx/Vy/Vz 速度指令，发送 Cxyzrx 在速度指令后立刻等待底盘回传，发送 Cxyz500rx 测试 DFLink 协议值 Vx=0.5 的线速度命令并读取回传，发送 Cxyz0 或 Cstop0 发零速帧。Vx=0.5 是已验证的手动/诊断速度；当前自动循线最高档为 Vx=1.0。线速度标定比例 协议值:真实速度 = 10:1 只用于后续估算真实速度，不会在发送前缩放协议值。

Wi-Fi 热点默认关闭，以降低功耗。需要后备诊断或无线烧录时，可通过 BLE 发送 W1 启动名为 DCar-Liner-XXXXXX 的 Wi-Fi 热点，默认密码为 DCar-Liner123。电脑手动连接该热点后，可以访问 ESP32 的 HTTP 和 WebSocket 接口读取完整参数表、查看遥测状态，并执行 JSON 调试命令。

小程序调参走 BLE，只收发 Kp/Ki/Kd/速度档位 并保存到 NVS，不改固件；Wi-Fi 后备仅用于诊断（读取参数表、查看遥测、执行 JSON 调试命令）。无线 OTA 升级已移除，固件只能通过 USB 或网页串口烧录更新。

电脑端 BLE 自动测试工具位于 tools/ble_debug_tester.swift。运行后会自动扫描、连接、发送指定命令、读取响应，并把日志写入 logs/ble-debug-tests/。较长 JSON 遥测响应仍会通过 ble_chunk 分包返回，电脑端工具会拼包后解析。

http://192.168.4.1/ 只是 ESP32 上的备用诊断入口，页面只提供静态 API 链接。主调参入口是 DHelper 小程序 BLE 参数面板。

调试接口的精确地址是 /api/health、/api/schema、/api/params 和 /api/telemetry。如果手误输入 /api/scheme，当前固件也会兼容返回参数表。建议现场先连续打开几次 /api/health，确认 HTTP 服务和内存余量稳定，再打开更大的参数表和遥测接口。

/api/telemetry 中的 line_uart 字段会显示循迹模块串口请求次数、收到的帧数、数字帧数、无效帧数、超时次数和最后一条原始帧，便于排查黑白线变化没有反应的问题。

车端采用停车调参策略：查看遥测不会影响小车状态；进入调参时固件会先停车并进入 PARAM_TUNING，只有在这个状态下才能修改、保存或恢复参数。退出调参后回到 SAFE_IDLE，不会自动继续行驶。

自动循迹需要两步明确指令：先进入 AUTO_ARMED，再进入 AUTO_RUNNING。上电默认是 SAFE_IDLE，底盘不动；丢线会进入扩幅摆扫找线，短暂传感器异常会进入保守找线动作，不会立刻故障停车；BLE 连接中断会立即发送停止并回到 SAFE_IDLE。

丢线恢复使用速度指令分段摆扫：线速度为 0，Z 角度增量使用当前档位正常转向上限，按左 0.8 s、右 1.6 s、左 3.2 s、右 6.4 s、左 12.8 s、右 25.6 s、继续右转 30.0 s 的序列扩大找线范围；最后一段仍未找回时停止并退出自动循线，回到 SAFE_IDLE。找回线后，telemetry 的 controller.recovery.relation 会记录 UNDERSTEER、OSCILLATION_SKEW 或 NO_REFERENCE，controller.recovery.stage 会显示 SWEEP、ONE_WAY_SEARCH 或 FAILED，segment_index、elapsed_ms 和 target_ms 会显示当前找线分段进度；这些记录当前只用于观察，不把几何判断反向绑定到 PID 或速度策略。

电脑端已经准备了一个离线状态机测试工具：在电脑连上 ESP32 热点后，在工程目录运行 python3 tools/state_machine_tester.py。它会先发送一次安全停车，让重复测试回到待机状态，然后自动访问 HTTP 和 WebSocket 接口，测试调参锁、手动低速测试、准备自动循迹和停车，并把日志写入 logs/state-machine-tests/。默认流程不会发送 start_auto；只有确认小车物理安全时，才使用 --include-auto 测试自动循迹运行状态。

当前固件已经按 ESP-IDF 官方示例和常见 ESP32 Wi-Fi 配置项目的做法加固 HTTP 调试服务：大 JSON 响应不再放在 HTTP 任务栈上，普通 API 响应会关闭短连接并禁止缓存，同时允许电脑端网页工具跨源访问。

DHelper 小程序 line-tuning 的短帧发送会一次性写入并保存 Kp/Ki/Kd/速度档位 四个值。JSON 调试接口仍保持停车调参规则，save_params 执行时才写入 NVS，避免高频调参损耗 flash。
