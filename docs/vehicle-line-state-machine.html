一句话答案

当前实现
1
巡线时不是“前进、左移、右移、旋转”这些动作状态。 这些是策略输出给底盘的动作命令，不是主状态。
2
巡线内部现在按 phase 和 offset 连续控制。 有线就进入 TRACK_LINE，再用 offset 算速度和转向；没线就进入 LINE_LOST 搜索。
3
连续 N 次图案判定目前没有真正接入。 主干已把单帧几何解析放入 line_interpreter，但还没有用连续 N 帧做趋势状态转换。

整车主状态机

这是车端最高层的状态机，位于 vehicle_state 模块。它决定现在能不能调参、能不能手动测试、能不能开始自动巡线、是否处于 OTA 或故障。

控制循环数据流

自动巡线不是“开始后发一次速度就完事”，而是控制任务按周期重复读取传感器、计算策略、发送底盘命令。

1. 读取整车状态 vehicle_state_get_snapshot 判断是 disabled、manual、armed 还是 running。
2. 读取巡线模块 非手动、非 OTA 时读取串口帧，超时则进入 sensor fault 输入。
3. 读取参数 档位映射线速度和最大转向，读取 PID 和积分限幅。
4. 策略计算 line_trace_policy_step 选择 phase 并计算底盘速度。
5. 输出到底盘 需要运动则发速度；需要停止则发 stop；失败进入故障。

AUTO_RUNNING 内部巡线 phase 机

这才是你重点关心的“行驶状态中的状态机”。当前版本的内部状态不是按每一种灯型细分，而是按运行条件进入不同 phase。

TRACK_LINE 的计算策略

当前主干的核心不是离散灯型分类，而是把八路传感器压成 bits、offset、active_count，再连续计算速度和转向。

输入八路数字值

巡线模块返回 $D,x1:...,x8:...#。当前主干把值为 0 的探头视为压到黑线，并置入 bits。

位置offset

八个探头权重为 -7,-5,-3,-1,+1,+3,+5,+7。多个探头有效时取平均，所以中间 4、5 同时有效时 offset 接近 0。

质量line quality

有效探头数量为 1 到 3 时质量 100；4 到 5 时质量 60；大于 5 时质量 30；没有有效探头时质量 0。

你说的“模式识别机”现在有没有？

如果把你的设想翻译成 MBD 语言，它应该是一个介于传感器解析和 PID 控制之间的“line pattern classifier”。主干现在已经有 line_interpreter 作为入口，但只做单帧轻量分类，还没有做连续 N 帧趋势状态机。

已有：phase + 单帧几何分类

代码会判断 DISABLED、AUTO_ARMED、AUTO_RUNNING、bits == 0、传感器超时等条件，并进入 STOPPED、TRACK_LINE、LINE_LOST、SENSOR_FAULT 等 phase。line_interpreter 还会输出 CENTERED、LEFT_OFFSET、RIGHT_OFFSET、LOST、SENSOR_FAULT 这些当前帧几何状态。

没有：连续 N 帧灯型状态机

当前没有 CENTER_STABLE、LEFT_DRIFT、RIGHT_DRIFT、SHARP_LEFT、INTERSECTION、RECOVERING 这类由历史时间窗口确认的状态。

如果下一版要做成真正 MBD 式策略机

我建议不替换现有 PID，而是在它前面加一个很小的模式分类层。这样既保留当前稳定手感，又能解释“连续几帧发生了什么”。

建议新增line_pattern_state

  • CENTER_STABLE：连续 N 帧 offset 在 -1 到 +1。
  • LEFT_DEPARTING：连续 N 帧 offset 为负并增大。
  • RIGHT_DEPARTING：连续 N 帧 offset 为正并增大。
  • LINE_LOST_LEFT / LINE_LOST_RIGHT：根据丢线前趋势确认搜索方向。
  • WIDE_LINE / INTERSECTION：有效探头数过多时不按普通 PID 处理。

保持不变底盘输出接口

模式识别层只输出“当前线型状态”和建议增益系数；最终仍由现有 line_trace_policy_step 统一生成 linear_mm_s 和 angular_mdeg_s。这样对小程序、BLE、底盘协议都最小侵入。

模块 Goal Inputs Outputs Independent Test
line_sensor_uart 把串口帧转成当前帧位置 $D,x1..x8# bits、offset、updated_ms 喂入固定帧，检查 bits 和 offset。
line_interpreter 把当前帧转成可解释线型状态，后续承接连续帧趋势 最近 N 帧 bits/offset/count 线型状态、置信度、推荐速度系数 离线输入帧序列，验证状态转换。
line_trace_policy 统一生成底盘速度命令 run mode、sensor status、line sample、pattern state、PID 参数 linear_mm_s、angular_mdeg_s、phase 用固定输入检查速度、转向、搜索方向。
